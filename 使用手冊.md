## 🌟 系統概覽

TW AutoTrader 是一套 **專業級台股自動交易系統**，整合：

- **四種經典策略**：VWAP 偏離、均線交叉、布林反轉、突破交易
- **雙券商支援**：玉山證券 API（主力） + 凱基證券 API
- **投資組合模擬**：DCA 每月定額 / 一筆資金兩種模型
- **專業風險控管**：單筆風險、每日虧損上限、交易次數限制
- **即時監控**：Telegram 通知 + LINE Notify
- **低成本部署**：GCP 定時排程（~NT$50/月）、本地 Docker build + 快速部署

## ⚠️ 第三步：策略暖身所需最少 K 線數

所有策略都是使用「滑動視窗（rolling window）」計算技術指標。在餵入的歷史資料還不夠多時，指標會是無效值（NaN）或噪聲極大，此時產生的交易訊號**不可靠**。

### 各策略暖身門檻

| 策略 | ⚡ 程式能產出第一個訊號 | 📊 指標真正穩定可靠 |
|------|----------------------|-------------------|
| **VWAP 偏離** | 6 根 K 線（RSI 限制） | 20 根（Std 收斂） |
| **均線交叉** | 2 根 K 線（shift 限制） | 22 根（MA21 完整一輪） |
| **布林反轉** | 2 根 K 線 | 20 根（通道穩定） |
| **突破交易** | 2 根 K 線（shift 限制） | 21 根（Donchian 通道滿） |

### 範例解說

以 **均線交叉（ma_cross, fast_period=9, slow_period=21）** 為例：

```
K 線  1 ~ 21：   MA21 只有部分資料，數值不具代表性
K 線     22：    MA21 首次涵蓋完整 21 根，交叉判斷才可靠
           ╰── 在此之前產生的黃金/死亡交叉都是假訊號
```

以 **VWAP 偏離（vwap, rsi_period=5, sigma_mult=1.5）** 為例：

```
K 線  1 ~ 5：    RSI 尚未產出有效值（全為 NaN）
K 線      6：    RSI 首次有效
K 線  1 ~ 19：   Std 使用 min_periods=1，數值逐漸收斂
K 線     20：    Std 完全收斂，偏離判斷才真正可靠
```

### 實戰建議

| 情境 | 建議餵入資料量 |
|------|--------------|
| **回測**（`backtest.py`） | 回測期間設為 **1 年以上**（約 250 根交易日 K 線），前 22 根視為 warm-up 即可 |
| **實盤剛啟動**（`live_trader_finmind.py`） | 程式會自動抓取 **30 天歷史資料**（約 22 根交易日 K 線），足夠所有策略暖身 |
| **多股實盤剛啟動**（`live_trader_multi.py`） | 同上，自動抓取 **30 天歷史資料** |
| **手動餵資料測試** | 至少餵 **22 根 K 線** 再開始觀察訊號 |

### 為什麼回測一定要設 `--start` 夠早？

```bash
# ❌ 錯誤：只給一個月資料，MA21 連一次交叉都算不準
python backtest.py --strategy ma_cross --start 2026-05-01

# ✅ 正確：至少給一年，前 22 根 warm-up 後還有 200+ 根可交易
python backtest.py --strategy ma_cross --start 2025-01-01
```

> 💡 **簡單記法：不管用哪個策略，餵少於 22 根 K 線就開始看訊號，跟擲硬幣沒兩樣。**

## 🚀 第四步：實盤交易

### 4.1 券商選擇（BROKER）

在 `.env` 中設定 `BROKER=kgi`（預設）或 `BROKER=esun`：

| BROKER | 行情資料源 | 下單券商 | 備註 |
|--------|-----------|---------|------|
| `kgi` + `USE_REAL_API=false` | KGI 模擬器 | KGI 模擬 | 開發測試用 |
| `kgi` + `USE_REAL_API=true` | KGI 即時 | 凱基證券 | 正式交易 |
| `esun` | 玉山行情 API（真資料） | 玉山證券 | USE_REAL_API 自動為 true |

#### 玉山 API 初次設定

`esun_sdk/` 目錄內已有以下檔案，專案會自動偵測：
- `config.simulation.ini.example` — 設定檔（含 API key/secret/account）
- `K12......_20270609.p12` — 憑證檔

只需完成密碼設定即可（二選一）：

**方式一：`.env` 填入密碼（自動寫入 keyring）**
```env
BROKER=esun
ESUN_ACCOUNT_PASSWORD=你的交易密碼
ESUN_CERT_PASSWORD=你的憑證密碼
```

**方式二：互動式設定**
```bash
python -c "from data.esun_provider import EsunProvider; EsunProvider.interactive_setup()"
```

注意事項：
- 密碼儲存在系統 keyring 中，不會顯示在程式碼中
- Docker 環境需設定環境變數 `PYTHON_KEYRING_BACKEND=keyrings.cryptfile.cryptfile.CryptFileKeyring` 與 `KEYRING_CRYPTFILE_PASSWORD`

### 4.2 單股交易（`live_trader_finmind.py`）

```bash
# 凱基模擬（開發測試）
python live_trader_finmind.py --symbol 2330 --strategy vwap

# 凱基真實交易
# 編輯 .env：BROKER=kgi, USE_REAL_API=true
python live_trader_finmind.py --symbol 2330 --strategy vwap
```

### 4.3 多股多策略分流（`live_trader_multi.py`）

同時監控多檔股票，每檔搭配不同策略：

```bash
# 使用 .env 中的 PORTFOLIO 設定
python live_trader_multi.py
```

`.env` 設定範例：
```env
PORTFOLIO=0050:bollinger,2330:ma_cross,2382:breakout,2881:vwap
```

每檔股票會獨立判斷，不會互相干擾。

### 4.4 股票數量上限建議

由於 `live_trader_multi.py` 採用**順序迴圈架構**（一支股票處理完才處理下一支），股票數量過多會導致每輪循環時間拉長：

| 股票數 | 每輪循環時間 | 建議 |
|--------|------------|------|
| 1~4 支 | ~68 秒 | ✅ 最佳，當前設計目標 |
| 5~10 支 | ~80 秒 | ✅ 良好，仍在 1 分鐘內 |
| 10~15 支 | ~95 秒 | ⚠️ 可接受，留意循環漂移 |
| 15+ 支 | 超過 100 秒 | ❌ 不建議，訊號失去即時性 |

**建議上限：15 支股票**。如果你需要同時監控超過 15 支股票，建議：
- 方案一：將程式改為非同步並行（asyncio）架構，同時發送 API 請求
- 方案二：開兩台 GCP e2-micro VM，各負責一半股票
- 方案三：改用 Cloud Run Jobs + Cloud Scheduler，每分鐘觸發一次（無主機管理）

> 💡 超過 15 支時程式會顯示警告訊息，但不會強制停止。

### 4.5 交易時段行為

程式會自動判斷台股交易時段，不同時間有不同的行為：

| 時段 | 時間 | 行為 |
|------|------|------|
| **盤中** | 週一~五 08:45-13:30 | 🔄 提前 15 分鐘暖機，09:00 開盤就能第一時間反應訊號 |
| **收盤** | 13:30-13:45 | ⏸️ 停止交易，等待 13:45 發送日報 + 產生儀表板 |
| **休眠** | 其餘時間（盤前、盤後、週末） | 💤 自動計算下次開盤時間，休眠到那時 |

> 💡 休眠期間還是會每小時醒來檢查一次，確保跨日、連假後也能準時恢復交易。系統在 08:45 提前暖機（建立連線、抓取開盤前資料），09:00 開盤即可立即反應。VM 的 GCP 排程（08:00-14:00）負責開關機，程式內的時間判斷則確保只在盤中交易。

## 🛡️ 第五步：風險控管

### 5.1 風險參數設定
在 `.env` 檔案中設定：

| 參數 | 預設值 | 說明 |
|------|--------|------|
| `INITIAL_CAPITAL` | `500000` | 初始資金（台幣） |
| `MAX_RISK_PER_TRADE` | `0.02` | 單筆交易最大風險（2%） |
| `MAX_DAILY_LOSS` | `0.03` | 每日最大可接受虧損（3%） |
| `MAX_DAILY_TRADES` | `5` | 每日最大交易次數 |

### 5.2 風險控管功能
- **單筆風險限制**：動態計算部位大小
- **每日虧損上限**：達上限後停止交易
- **交易次數限制**：避免過度交易
- **漲跌停過濾**：跳過流動性枯竭標的

### 5.3 部位計算邏輯
```
風險金額 = 初始資金 × MAX_RISK_PER_TRADE
止損距離 = 股價 × 2%（固定百分比）
股數 = 風險金額 / 止損距離
張數 = floor(股數 / 1000)  # 台股一張 = 1000 股
```

### 5.4 每月預算控管（DCA 每月定額模式）

`live_trader_multi.py` 支援**每月每策略預算上限**，適用於每月 NT$20,000 定期定額配置：

```env
# 各策略每月預算（0 = 不限制）
MONTHLY_BUDGET_BOLLINGER=10000      # ETF 逆勢策略，預算最高
MONTHLY_BUDGET_VWAP=3000            # 金融存股策略
MONTHLY_BUDGET_MA_CROSS=4000        # 權值股順勢策略
MONTHLY_BUDGET_BREAKOUT=3000        # 飆股順勢策略
```

> 💡 **NT$500,000 一筆資金模式**：將 `MONTHLY_BUDGET_*` 全設為 `0`（不限制），改由 `TOTAL_CAPITAL + ALLOC_*` 百分比機制控管總量上限（見 §5.5）。

運作方式：
- **只有買進（BUY）會扣預算**，賣出（SELL）不計入
- 預算以「買進成本」（股數 × 成交價）計算，**不是**以每月總額預扣
- 月中達到預算上限後，該策略就不再買進，直到下個月 1 號重置
- 設定 `0` 或留空 = 不限制預算
- 預算追蹤資料儲存在 `logs/monthly_budget.json`

可搭配風險控管同時使用，兩層防護互不衝突。

### 5.5 策略資金配置上限

為避免單一策略超額消耗資金，系統支援**總資金 × 百分比**的上限控管：

```env
TOTAL_CAPITAL=500000       # 可動用總資金（預設 = INITIAL_CAPITAL）
ALLOC_BOLLINGER=40         # 布林通道反轉 40%
ALLOC_VWAP=20              # VWAP 偏離反轉 20%
ALLOC_MA_CROSS=25          # 均線交叉 25%
ALLOC_BREAKOUT=15          # 唐奇安突破 15%
```

運作方式：
- **TOTAL_CAPITAL** 可與 INITIAL_CAPITAL 不同（例如還有其他資金來源）
- 各策略累計買進成本達上限後，該策略跳過新買進訊號（賣出後釋放額度，可再次買進）
- 賣出時以平均成本釋放額度：`釋放額度 = 賣出股數 × (累計買進成本 ÷ 累計買進股數)`
- 追蹤資料儲存在 `logs/strategy_allocation.json`（程式重啟後沿用）
- 不設定則各策略有預設百分比（40/20/20/20）

> 💡 每月定額 DCA 模式建議同時設定 `MONTHLY_BUDGET_*`（§5.4）與 `ALLOC_*`，雙層防護。

### 5.6 大盤年線過濾

避免在空頭市場（大盤跌破年線）時逆勢買進：

- 啟動時從 **FinMind** 抓取加權指數（TX00）日線，計算 MA200
- 若指數 < MA200，跳過所有買進訊號
- 若 FinMind 抓取失敗（網路、API 問題），**安全跳過不過濾**，不影響正常交易
- 設定 `MARKET_TREND_FILTER=false` 可關閉

```env
MARKET_TREND_FILTER=true
```

### 5.7 金字塔加碼（Bollinger 分批進場）

Bollinger 逆勢策略觸發買進時，分批次進場降低成本，而非一次性買足：

| 梯次 | 條件 | 買入股數 |
|------|------|---------|
| Tier 1 | 首次觸發訊號 | 1,000 股 |
| Tier 2 | 價格再跌 3% | 2,000 股 |
| Tier 3 | 價格再跌 5% | 3,000 股 |

僅限 Bollinger 策略，可透過 `.env` 啟用：

```env
PYRAMID_ENABLED=true
PYRAMID_TIER1_SHARES=1000
PYRAMID_TIER2_SHARES=2000
PYRAMID_TIER2_DROP=0.03
```

> ⚠️ 金字塔加碼適用於 NT$500,000 資金規模。小資每月 NT$20,000 建議使用預設較低股數。

## 🛠️ 故障排除

### 常見問題與解決方案

| 問題 | 解決方案 |
|------|----------|
| **`Failed to get ticker '2330.TW'`** | 改用 FinMind 資料源，確保 `.env` 有 `FINMIND_API_TOKEN` |
| **凱基 API 連接失敗** | 確認 `KGI_API_KEY` 和 `KGI_API_SECRET` 正確，檢查網路連線 |
| **玉山 API 連接失敗** | 確認 `ESUN_ENTRY`、憑證路徑正確；執行 `interactive_setup()` 重新設定密碼 |
| **玉山 login 報錯** | 檢查 keyring 中帳號/憑證密碼是否正確；可設 `ESUN_ACCOUNT_PASSWORD` 與 `ESUN_CERT_PASSWORD` 重新執行 |
| **`ModuleNotFoundError: No module named 'esun_marketdata'`** | 先安裝 SDK `.whl` 檔：`pip install esun_sdk/esun_marketdata-*.whl esun_sdk/esun_trade-*.whl` |
| **Telegram 未收到通知** | 驗證 `TELEGRAM_BOT_TOKEN` 和 `TELEGRAM_CHAT_ID`，檢查 Bot 是否被封鎖 |
| **無交易訊號** | 延長回測期間，或調整策略參數（如降低 `sigma_mult`） |
| **風險控管攔截所有交易** | 檢查 `MAX_DAILY_LOSS` 是否過低，或手動清除 `logs/performance.csv` |

## 🔒 安全注意事項

1. **API Key 保護**：
   - 切勿將 `.env` 檔案上傳到 GitHub
   - 在券商後台設定 IP 限制（如果支援）
   - 注意 Opencode 的 `auth.json` 中也存有 API Token（位置：`~/.local/share/opencode/auth.json` 或專案內 `.opencode/` 下）
   - `~/.bashrc` 中也可能設有環境變數形式的 API Key

2. **WSL 備份／移機注意**：
   WSL 的 `.tar` 匯出會**完整打包**以下明碼憑證，請勿將備份檔上傳到公有雲或不信任的存放空間：
   - `~/tw-autotrader/.env` — 玉山交易密碼、Telegram Bot Token、FinMind Token
   - `~/.config/opencode/auth.json` — Opencode API Key
   - `~/.bashrc` — 可能寫死的環境變數
   - `~/.ssh/` — SSH 私鑰
  
   🔐 **建議**：移機前先將上述檔案中真實的 Key 換成假的，移完再手動填回。

3. **資金管理**：
   - 初始資金建議不超過總資產 10%
   - 先用模擬模式測試至少 1 週

4. **監控機制**：
   - 每日檢查交易日誌
   - 設定 Telegram 通知確保即時掌握
