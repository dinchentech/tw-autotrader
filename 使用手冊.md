## 🆕 V2.0：用戶自訂策略

從 V2.0 開始，您可以在不修改加密主程式 `live_trader_multi.py` 的情況下，自由定義和使用自己的交易策略。

**快速連結：**
- ⚙️ [用戶自訂策略快速入門](用戶自訂策略快速入門.md)
- 📝 [用戶自訂策略使用說明](用戶自訂策略使用說明.md)
- 📋 [V2.0 更新說明](CHANGELOG_V2.0.md)

**四組備用策略：**
| 策略代號 | 名稱 | 預設邏輯 |
|----------|------|----------|
| `g1_strategy_1` | 均線交叉 | 快線穿過慢線向上買進 |
| `g1_strategy_2` | RSI 超買超賣 | RSI<30 買進, RSI>70 賣出 |
| `g2_strategy_1` | 價格動能 | N 日漲跌幅超門檻買進 |
| `g2_strategy_2` | 量價配合 | 價格站穩均線+量放大買進 |

**使用範例：**
```bash
# 測試策略載入
python test_user_strategies.py

# 回測自訂策略
python backtest.py --strategy g1_strategy_1 --symbol 2330

# 在 .env 中使用
PC_2330={"strategy":"g1_strategy_1","alloc":20,"fast_period":10,"slow_period":20}
```

## 🌟 系統概覽

TW AutoTrader 是一套 **專業級台股自動交易系統**，整合：

- **四種經典策略**：VWAP 偏離、均線交叉、布林反轉、突破交易 + 簡單持有keep_wait及法人動能  兩種策略共6種策略
- **雙券商支援**：玉山證券 API（主力） + 凱基證券 API
- **投資組合模擬**：DCA 每月定額 / 一筆資金兩種模型
- **專業風險控管**：單筆風險、每日虧損上限、交易次數限制
- **即時監控**：Telegram 通知
- **低成本部署**：GCP 定時排程（~NT$50/月）、本地 Docker build + 快速部署

## 🔧 第一步：環境設定

### 1.1 系統需求
- **作業系統**：Windows / macOS / Linux
- **Python 版本**：3.8 或更高（推薦 3.9）
- **網路環境**：穩定網路連線

### 1.2 安裝依賴(使用wsl備份可跳過只要於tw-autotrader目錄中執行git pull即可)
```bash
# 克隆專案
git clone https://github.com/dinchentech/tw-autotrader.git
cd tw-autotrader

# 安裝依賴
pip install -r requirements.txt
pip install python-dotenv
```

### 1.3 設定環境變數
```bash
# 選擇你的方案，複製對應的環境範例

#### `.env` 檔案範例：

完整的 `.env` 範例請見專案中的 [`.env.example.txt`](.env.example.txt)，此處扼要說明核心區塊。

**系統級設定**（券商、風險控管、通知）：

USE_REAL_API=false                        # false=KGI mock, true=玉山 SDK（行情+交易）

# DCA 模式設定（0 = 一筆資金 Lumpsum，>0 = 每月定期定額 DCA）

# 通知
TELEGRAM_BOT_TOKEN=your_telegram_bot_token
TELEGRAM_CHAT_ID=your_telegram_chat_id
```

**個股設定（V1.11 格式）** — 每檔股票使用 `PC_<代號>` JSON 變數：
```env
PC_0050={"strategy":"bollinger","alloc":35,"window":20,"std_dev":2.0,"rsi_period":5}

### 1.5 外部資金追蹤（capital.txt）

V1.2 新增 `capital.txt` 檔案，追蹤外部資金投入（加碼/提領），支援模擬與實盤。

```
capital.txt 格式：
金額, YYYY/MM/DD  # comment

範例：
100000, 2023/06/01  # 外部加碼 10 萬
-50000, 2024/01/15  # 提領 5 萬
200000, 2024/06/01  # 年中加碼 20 萬
```

- 模擬模式：`simulate_portfolio.py` 每日自動讀取，新增紀錄觸發資金調整
- 實盤模式：`live_trader_multi.py` 每日檢查，新增資金自動分配至 keep_wait 策略（70% 立即買入）
- 追蹤紀錄：`logs/processed_capital.json` 記錄已處理項目，避免重複處理

### 1.6 DCA 模式切換（DCA_AMOUNT）

`.env` 新增 `DCA_AMOUNT` 參數，自動判斷 DCA/Lumpsum 模式：

| 值 | 模式 | 說明 |
|---|---|---|
| `0` | Lumpsum | 一筆資金（預設 `TOTAL_CAPITAL` 金額） |
| `>0` | DCA | 每月定期定額（值 = 每月投入金額） |

```bash
# Lumpsum 模式（一筆 50 萬）
DCA_AMOUNT=0
TOTAL_CAPITAL=500000

# DCA 模式（每月 2 萬）
DCA_AMOUNT=20000
```

模擬時自動選擇模式，無需手動指定 `--mode`。

## 📊 第二步：回測驗證

### 2.1 執行 Yahoo Finance 回測（快速批次驗證）

```bash
# 使用預設策略（vwap）回測全部標的
python backtest.py

# 指定策略與開始日期
python backtest.py --strategy ma_cross --start 2022-01-01

# 調整策略參數（免改程式碼！）
python backtest.py --strategy ma_cross --fast_period 5 --slow_period 30
python backtest.py --strategy bollinger --std_dev 2.5 --rsi_period 7
python backtest.py --strategy breakout --lookback 40
python backtest.py --strategy vwap --sigma_mult 2.0
```

### 1.4 V1.11 個股獨立參數設定（PC_ 格式）

V1.11 引入 `PC_<代號>` JSON 格式，將**每檔股票的策略、資金配置、策略參數全部封裝在單一變數中**，不再使用舊式的全域變數。

#### 語法規則

```
PC_<股票代號>={"strategy":"策略名稱","alloc":資金佔比%, ...策略參數...}
```

- **PC_** 前綴：所有個股設定以此開頭，程式自動掃描
- **JSON 內容**：單行 JSON，支援以下欄位

#### 通用欄位（每支股票必填）

| 欄位 | 型態 | 說明 |
|------|------|------|
| `strategy` | 字串 | 策略名稱：`bollinger` / `vwap` / `ma_cross` / `breakout` / `keep_wait` |
| `alloc` | 數字 | 資金配置百分比（相對於 `TOTAL_CAPITAL`） |
| `position_amount` | 數字 | 單筆買進金額（元），bollinger/vwap/ma_cross 適用 |
| `monthly_budget` | 數字 | 每月預算上限（0 = 不限制） |

#### 策略特定參數

各策略可接受的參數名稱（依 `core/config_loader.py` 中的 `STRATEGY_PARAM_KEYS`）：

| 策略 | 可用參數 |
|------|---------|
| **bollinger** | `window`, `std_dev`, `rsi_period` |
| **vwap** | `sigma_mult`, `rsi_period` |
| **ma_cross** | `fast_period`, `slow_period`, `atr_threshold` |
| **breakout** | `lookback`, `atr_period`, `atr_threshold` |
| **keep_wait** | 無（使用下方 DCA 低接參數） |

#### 金字塔加碼（限 bollinger 策略）

| 參數 | 預設 | 說明 |
|------|------|------|
| `pyramid_enabled` | `false` | 啟用分批次進場 |
| `pyramid_tier1_shares` | `1000` | Tier 1 首次買入股數 |
| `pyramid_tier2_shares` | `2000` | Tier 2 加碼股數 |
| `pyramid_tier3_shares` | `3000` | Tier 3 加碼股數 |
| `pyramid_tier2_drop` | `0.03` | Tier 2 觸發跌幅（3%） |
| `pyramid_tier3_drop` | `0.05` | Tier 3 觸發跌幅（5%） |

#### keep_wait 低接參數（V1.2）

| 參數 | 預設 | 說明 |
|------|------|------|
| `initial_buy_pct` | `0.7` | 首次進場資金比例（70% 立即買入，30% 留作加碼） |
| `add_drop_pct` | `5` | 加碼觸發跌幅（%） |
| `add_shares` | `6` | 每次加碼股數 |
| `max_additions` | `2` | 最多加碼次數（設 `0` 可關閉 DCA，完全不加碼） |
| `max_entry_price` | `0` | 進場價格上限（設 `0` 不檢查，高於此價格跳過買入） |
| `tp_trigger_pct` | `15` | 停利觸發漲幅（%）（未設 `tp_tiers` 時使用） |
| `tp_sell_ratio` | `50` | 停利賣出比例（%）（未設 `tp_tiers` 時使用） |
| `cooldown_days` | `30` | 賣出後冷卻天數 |

> 💡 **V1.2 改進**：Lumpsum 模式下，keep_wait 首次進場不再只買 12 股（~NT$894），而是投入 70% 分配資金（NT$35,000），解決資金閒置問題。績效從 +3.7% 提升至 +102.4%。

#### 多層停利參數 `tp_tiers`（V2.06）

取代單一的 `tp_trigger_pct` / `tp_sell_ratio`，支援分批停利：

```
tp_tiers=[{"pct":5,"ratio":25},{"pct":10,"ratio":25},{"pct":15,"ratio":100}]
```

| 層級 | pct（漲幅門檻） | ratio（賣出比例） | 行為 |
|------|----------------|------------------|------|
| Tier 1 | `5` | `25` | 漲 5% 時賣出 25% 持股 |
| Tier 2 | `10` | `25` | 漲 10% 時再賣出 25% |
| Tier 3（最後一層） | `15` | `100` | 漲 15% 時出清剩餘全部 |

**規則**：
- 各層依序觸發，一次只觸發一層
- 最後一層不論 ratio 設定多少，一律出清全部剩餘持股
- 設 `tp_tiers` 時，`tp_trigger_pct` / `tp_sell_ratio` 被忽略
- 未設 `tp_tiers` 時，沿用舊有單一停利邏輯（向下相容）

**範例**：三檔反向 ETF 各 10 萬，三層停利，不 DCA 加碼

```env
PC_00632R={"strategy":"keep_wait","alloc":11,"initial_buy_pct":1.0,"max_additions":0,
           "tp_tiers":[{"pct":5,"ratio":25},{"pct":10,"ratio":25},{"pct":15,"ratio":100}]}
PC_00664R={"strategy":"keep_wait","alloc":11,"initial_buy_pct":1.0,"max_additions":0,
           "tp_tiers":[{"pct":5,"ratio":25},{"pct":10,"ratio":25},{"pct":15,"ratio":100}]}
PC_00676R={"strategy":"keep_wait","alloc":11,"initial_buy_pct":1.0,"max_additions":0,
           "tp_tiers":[{"pct":5,"ratio":25},{"pct":10,"ratio":25},{"pct":15,"ratio":100}]}
```

> 💡 **V2.06 新增**：`tp_tiers` 適用於需要分批獲利了結的情境（如反向 ETF 避險、波段操作），`max_additions=0` 可完全關閉 DCA 加碼，適合只想一次性持有、不打算攤平的標的。

#### 實例對照

**舊格式（V1.0）：**
```env

<a id="第三步策略暖身"></a>
## ⚠️ 第三步：策略暖身所需最少 K 線數

所有策略都是使用「滑動視窗（rolling window）」計算技術指標。在餵入的歷史資料還不夠多時，指標會是無效值（NaN）或噪聲極大，此時產生的交易訊號**不可靠**。

### 各策略暖身門檻

| 策略 | ⚡ 程式能產出第一個訊號 | 📊 指標真正穩定可靠 |
|------|----------------------|-------------------|
| **VWAP 偏離** | 6 根 K 線（RSI 限制） | 20 根（Std 收斂） |
| **均線交叉** | 2 根 K 線（shift 限制） | 22 根（MA21 完整一輪） |
| **布林反轉** | 2 根 K 線 | 20 根（通道穩定） |
| **突破交易** | 2 根 K 線（shift 限制） | 21 根（Donchian 通道滿） |
| **法人動能** | 30 根（低吃分數計算） | 60 根（低吃回溯期） |

### 範例解說

以 **均線交叉（ma_cross, fast_period=9, slow_period=21）** 為例：

```
K 線  1 ~ 21：   MA21 只有部分資料，數值不具代表性
K 線     22：    MA21 首次涵蓋完整 21 根，交叉判斷才可靠
           ╰── 在此之前產生的黃金/死亡交叉都是假訊號
```

以 **VWAP 偏離（vwap, rsi_period=5, sigma_mult=1.5）** 為例：

```
K 線  1 ~ 5：    RSI 尚未產出有效值（全為 NaN）
K 線      6：    RSI 首次有效
K 線  1 ~ 19：   Std 使用 min_periods=1，數值逐漸收斂
K 線     20：    Std 完全收斂，偏離判斷才真正可靠
```

### 實戰建議

| 情境 | 建議餵入資料量 |
|------|--------------|
| **回測**（`backtest.py`） | 回測期間設為 **1 年以上**（約 250 根交易日 K 線），前 22 根視為 warm-up 即可 |
| **實盤剛啟動**（`live_trader_finmind.py`） | 程式會自動抓取 **30 天歷史資料**（約 22 根交易日 K 線），足夠所有策略暖身 |
| **多股實盤剛啟動**（`live_trader_multi.py`） | 同上，自動抓取 **30 天歷史資料** |
| **手動餵資料測試** | 至少餵 **22 根 K 線** 再開始觀察訊號 |

### 為什麼回測一定要設 `--start` 夠早？

```bash
# ❌ 錯誤：只給一個月資料，MA21 連一次交叉都算不準
python backtest.py --strategy ma_cross --start 2026-05-01

# ✅ 正確：至少給一年，前 22 根 warm-up 後還有 200+ 根可交易
python backtest.py --strategy ma_cross --start 2025-01-01
```

> 💡 **簡單記法：不管用哪個策略，餵少於 22 根 K 線就開始看訊號，跟擲硬幣沒兩樣。**

## 🚀 第四步：實盤交易

### 4.1 券商選擇（BROKER）

在 `.env` 中設定 `BROKER=kgi`（預設）或 `BROKER=esun`：

| BROKER | 行情資料源 | 下單券商 | 備註 |
|--------|-----------|---------|------|
| `kgi` + `USE_REAL_API=false` | KGI 模擬器 | KGI 模擬 | 開發測試用 |
| `kgi` + `USE_REAL_API=true` | KGI 即時 | 凱基證券 | 正式交易 |
| `esun` | 玉山行情 API（真資料） | 玉山證券 | USE_REAL_API 自動為 true |

#### 玉山 API 環境切換：模擬 → 真實

玉山 SDK 有模擬／真實兩套環境，使用**不同的 API Endpoint 和不同的 API Key/Secret**。

切換時**不需要改程式碼**，只需準備兩份設定檔並透過 `.env` 指定用哪一份：

| 設定檔 | 用途 | Entry URL |
|--------|------|-----------|
| `esun_sdk/config.simulation.ini` | 模擬環境（沙箱，不會真的成交） | `https://esuntradingapi-simulation.esunsec.com.tw/api/v1` |
| `esun_sdk/config.ini` | 真實環境（送單到交易所） | `https://esuntradingapi.esunsec.com.tw/api/v1` |

**模擬環境**（預設）— 專案開箱即用：

程式會自動讀取 `esun_sdk/config.simulation.ini`，不需額外設定。`.env` 中只需：

```env
BROKER=esun
```

**切換到真實環境** — 三步驟：

① 建立 `esun_sdk/config.ini`（向玉山申請開通後會拿到真實環境的 Key/Secret）：

```ini
[Core]
Entry = https://esuntradingapi.esunsec.com.tw/api/v1
[Cert]
Path = ;esun_sdk/你的憑證檔.p12
[Api]
Key = <真實環境的 API Key>
Secret = <真實環境的 API Secret>
[User]
Account = 你的帳號
```

> ⚠️ 真實盤的 `.ini` **不需要** `Environment = REAL` 這行。玉山 SDK 依 Entry URL 自動判別環境。

② `.env` 加一行，告訴程式用真實盤的設定檔：

```env
ESUN_CONFIG_INI=esun_sdk/config.ini
```

③ 確認 `.env` 密碼設定正確（模擬和真實用同一組憑證密碼）：

```env
ESUN_ACCOUNT_PASSWORD=你的交易密碼
ESUN_CERT_PASSWORD=你的憑證密碼
```

切回模擬環境只需把 `ESUN_CONFIG_INI` 那行註解掉或刪除，程式會自動 fallback 到 `config.simulation.ini`。

> ⚠️ **IP 白名單**：每個 API Key（模擬／真實分開）都有獨立的 IP 白名單。登入 [E.Sun 交易 API 管理後台](https://esuntradingapi.esunsec.com.tw/) 設定允許連線的 IP。如果收到 `AGA0002: Invalid IP` 錯誤，代表伺服器 IP 不在白名單中。
>
> ⚠️ **`.env` 和 `*.ini` 不會上傳 git**（`.gitignore` 已排除），金鑰和密碼安全無虞。

#### 玉山 API 初次設定

### 4.4 股票數量上限建議

由於 `live_trader_multi.py` 採用**順序迴圈架構**（一支股票處理完才處理下一支），股票數量過多會導致每輪循環時間拉長：

| 股票數 | 每輪循環時間 | 建議 |
|--------|------------|------|
| 1~4 支 | ~68 秒 | ✅ 最佳，當前設計目標 |
| 5~10 支 | ~80 秒 | ✅ 良好，仍在 1 分鐘內 |
| 10~15 支 | ~95 秒 | ⚠️ 可接受，留意循環漂移 |
| 15+ 支 | 超過 100 秒 | ❌ 不建議，訊號失去即時性 |

**建議上限：15 支股票**。如果你需要同時監控超過 15 支股票，建議：
- 方案一：將程式改為非同步並行（asyncio）架構，同時發送 API 請求
- 方案二：開兩台 GCP e2-micro VM，各負責一半股票
- 方案三：改用 Cloud Run Jobs + Cloud Scheduler，每分鐘觸發一次（無主機管理）

> 💡 超過 15 支時程式會顯示警告訊息，但不會強制停止。

### 4.5 交易時段行為

程式會自動判斷台股交易時段，不同時間有不同的行為：

| 時段 | 時間 | 行為 |
|------|------|------|
| **盤中** | 週一~五 08:45-13:30 | 🔄 提前 15 分鐘暖機，09:00 開盤就能第一時間反應訊號 |
| **收盤** | 13:30-13:45 | ⏸️ 停止交易，等待 13:45 發送日報 + 產生儀表板 |
| **休眠前** | 13:45 後首次進入休眠 | 💤 Telegram 發送持倉報告 + 更新儀表板網頁，然後休眠 |
| **休眠** | 其餘時間（盤前、盤後、週末） | 💤 自動計算下次開盤時間，休眠到那時 |

> 💡 1.休眠期間還是會每小時醒來檢查一次，確保跨日、連假後也能準時恢復交易。系統在 08:45 提前暖機（建立連線、抓取開盤前資料），09:00 開盤即可立即反應。VM 的 GCP 排程（08:00-14:00）負責開關機，程式內的時間判斷則確保只在盤中交易。
>    2.**v2.02 起**：13:45 發送日報後，當程式第一次進入長時間休眠時，會再次透過 Telegram 發送完整的庫存持倉報告（含各股成本均價、市價、未實現損益），並重新產生 `logs/dashboard.html`，確保 GitHub Pages 儀表板同步更新到收盤後的持股狀態。

<a id="第五步風險控管"></a>
## 🛡️ 第五步：風險控管

### 5.1 風險參數設定
在 `.env` 檔案中設定：

| 參數 | 預設值 | 說明 |
|------|--------|------|
| `INITIAL_CAPITAL` | `500000` | 初始資金（台幣） |
| `MAX_RISK_PER_TRADE` | `0.02` | 單筆交易最大風險（2%） |
| `MAX_DAILY_LOSS` | `0.03` | 每日最大可接受虧損（3%） |
| `MAX_DAILY_TRADES` | `5` | 每日總交易次數上限 |
| `MAX_DAILY_TRADES_PER_SYMBOL` | `1` | 每檔股票每日交易次數上限（0=無限制） |

### 5.2 風險控管功能
- **單筆風險限制**：動態計算部位大小
- **每日虧損上限**：達上限後停止交易
- **交易次數限制**：避免過度交易（全域 + 每檔每日上限）
- **漲跌停過濾**：跳過流動性枯竭標的

### 5.3 部位計算邏輯
```
風險金額 = 初始資金 × MAX_RISK_PER_TRADE
止損距離 = 股價 × 2%（固定百分比）
股數 = 風險金額 / 止損距離
張數 = floor(股數 / 1000)  # 台股一張 = 1000 股
```

### 5.4 每月預算控管（DCA 每月定額模式）

`live_trader_multi.py` 支援**每月每策略預算上限**，適用於每月 NT$20,000 定期定額配置：

```env
# 各策略每月預算（0 = 不限制）
MONTHLY_BUDGET_BOLLINGER=10000      # ETF 逆勢策略，預算最高
MONTHLY_BUDGET_VWAP=3000            # 金融存股策略
MONTHLY_BUDGET_MA_CROSS=4000        # 權值股順勢策略
MONTHLY_BUDGET_BREAKOUT=3000        # 飆股順勢策略
```

> 💡 **NT$500,000 一筆資金模式**：將 `MONTHLY_BUDGET_*` 全設為 `0`（不限制），改由 `TOTAL_CAPITAL + ALLOC_*` 百分比機制控管總量上限（見 §5.5）。

運作方式：
- **只有買進（BUY）會扣預算**，賣出（SELL）不計入
- 預算以「買進成本」（股數 × 成交價）計算，**不是**以每月總額預扣
- 月中達到預算上限後，該策略就不再買進，直到下個月 1 號重置
- 設定 `0` 或留空 = 不限制預算
- 預算追蹤資料儲存在 `logs/monthly_budget.json`

可搭配風險控管同時使用，兩層防護互不衝突。

### 5.5 策略資金配置上限

為避免單一策略超額消耗資金，系統支援**總資金 × 百分比**的上限控管：

```env
TOTAL_CAPITAL=500000       # 可動用總資金（預設 = INITIAL_CAPITAL）
ALLOC_BOLLINGER=40         # 布林通道反轉 40%
ALLOC_VWAP=20              # VWAP 偏離反轉 20%
ALLOC_MA_CROSS=25          # 均線交叉 25%
ALLOC_BREAKOUT=15          # 唐奇安突破 15%
```

運作方式：
- **TOTAL_CAPITAL** 可與 INITIAL_CAPITAL 不同（例如還有其他資金來源）
- 各策略累計買進成本達上限後，該策略跳過新買進訊號（賣出後釋放額度，可再次買進）
- 賣出時以平均成本釋放額度：`釋放額度 = 賣出股數 × (累計買進成本 ÷ 累計買進股數)`
- 追蹤資料儲存在 `logs/strategy_allocation.json`（程式重啟後沿用）
- 不設定則各策略有預設百分比（40/20/20/20）

> 💡 每月定額 DCA 模式建議同時設定 `MONTHLY_BUDGET_*`（§5.4）與 `ALLOC_*`，雙層防護。

### 5.6 大盤年線過濾

避免在空頭市場（大盤跌破年線）時逆勢買進：

- 啟動時從 **FinMind** 抓取加權指數（TX00）日線，計算 MA200
- 若指數 < MA200，跳過所有買進訊號
- 若 FinMind 抓取失敗（網路、API 問題），**安全跳過不過濾**，不影響正常交易
- 設定 `MARKET_TREND_FILTER=false` 可關閉

```env
MARKET_TREND_FILTER=true
```

### 5.7 金字塔加碼（Bollinger 分批進場）

Bollinger 逆勢策略觸發買進時，分批次進場降低成本，而非一次性買足：

| 梯次 | 條件 | 買入股數 |
|------|------|---------|
| Tier 1 | 首次觸發訊號 | 1,000 股 |
| Tier 2 | 價格再跌 3% | 2,000 股 |
| Tier 3 | 價格再跌 5% | 3,000 股 |

僅限 Bollinger 策略，可透過 `.env` 啟用：

```env
PYRAMID_ENABLED=true
PYRAMID_TIER1_SHARES=1000
PYRAMID_TIER2_SHARES=2000
PYRAMID_TIER2_DROP=0.03
```

> ⚠️ 金字塔加碼適用於 NT$500,000 資金規模。小資每月 NT$20,000 建議使用預設較低股數。

### 5.8 獲利滾入本金（Profit Rolling）

系統支援**定期將已實現獲利自動滾入本金**，讓資金複利成長，而非閒置在帳戶中。

#### 參數設定

在 `.env` 中設定：

| 參數 | 預設值 | 說明 |
|------|--------|------|
| `PROFIT_ROLL_MONTHS` | `5` | 每 N 個月滾入一次（0 = 每次賣出立即滾入） |
| `PROFIT_ROLL_PERCENTAGE` | `100` | 滾入比例（%），100 = 全部獲利滾入 |

```env
PROFIT_ROLL_MONTHS=5
PROFIT_ROLL_PERCENTAGE=100
```

> ⚠️ **重要說明**：
> - `PROFIT_ROLL_MONTHS` 是**時間間隔**（每幾個月滾入一次），**不是倍數**（不是「獲利乘以 N 倍」）。
> - `M=0` 表示**每次賣出立即滾入**已實現獲利，無等待期。
> - `M=5` 表示**每 5 個月滾入一次**（預設值），兼顧複利效果與風險控制。
> - 使用方式：`python simulate_portfolio.py --profit-roll-months 5 --profit-roll-percentage 1.0`

#### 運作機制

1. **觸發條件**：`M=0` 時每次賣出立即滾入；`M>0` 時每 `M` 個月檢查一次（從初始日期起算）
2. **滾入金額**：`已實現損益 × P%`（已賣出股票的淨利潤）
3. **資金來源**：僅使用帳戶內資金，不追加外部資金
4. **資金分配**：滾入的資金加入 `general_cash`，可投資所有策略標的

#### 範例說明

假設：`INITIAL_CAPITAL=500,000`、`PROFIT_ROLL_MONTHS=5`、`PROFIT_ROLL_PERCENTAGE=100`

| 時間 | 已實現獲利 | 滾入金額 | 新本金 |
|------|-----------|---------|--------|
| 第 5 個月 | +NT$50,000 | NT$50,000 | NT$550,000 |
| 第 10 個月 | +NT$80,000 | NT$80,000 | NT$630,000 |
| 第 15 個月 | +NT$120,000 | NT$120,000 | NT$750,000 |

> 💡 **複利效應**：每 5 個月滾入一次，資金穩定複利成長。長期下來報酬率差異可觀。
>
> ⚠️ **注意**：僅限已賣出（已實現）的獲利才會滾入。未賣出的浮動盈虧不計入。

## 💰 第九步：外部資金追蹤（capital.txt）

### 9.1 使用說明

`capital.txt` 是**回測與模擬專用的外部資金注入檔**，讓你可以在回測期間的特定日期「手動加碼或提領資金」，不需要改程式碼或 `.env`。

**格式**（一行一筆）：

```
金額, YYYY/MM/DD  # 註解（選擇性）
```

- **金額**：整數，正數為注入、負數為提領
- **日期**：該筆資金生效的交易日
- **註解**：接在 `#` 後面，僅供辨識用

**範例**：

```
100000, 2023/06/01  # 年中加碼 10 萬
-50000, 2024/01/15  # 提領 5 萬
20000,  2025/03/10  # 額外小額加碼
```

**支援的模擬模式**：

- `simulate_portfolio.py` 的 `--mode lumpsum`（一筆資金模式）
- 每月月底發放（與獲利滾入同期執行）
- 加碼金額會依各標的 `alloc` 權重自動分配至現金桶

**注意事項**：

1. `capital.txt` **僅影響回測/模擬**，不影響實盤 `live_trader_multi.py`
2. 回測報告中的「外部加碼」項目即來自此檔
3. 若檔案不存在或空的，模擬會跳過，不影響結果
4. 提領金額若超過可用現金，該筆會部分執行或跳過（視該日持倉現金而定）

| **Dashboard 有買/賣次數，但總交易次數是 0** | 見下方說明 ↓ |

> **Dashboard「有買/賣次數但交易次數是 0」的解釋：**
>
> 兩者的計算邏輯不同，不是 bug：
>
> | 欄位 | 來源 | 意義 |
> |------|------|------|
> | **買進次數 / 賣出次數** | CSV 原始行數 | 不管有沒有成交，CSV 有幾筆就算幾筆 |
> | **買進股數 / 賣出股數** | CSV 原始數量加總 | 同上，純加總 |
> | **總交易次數** | 只有 **BUY→SELL 配對成功（FIFO）** 才算 | 買入的股票賣掉了、算出損益，才算 1 次 |
>
> ### 常見情況
>
> 1. **剛開始跑，只有買進還沒賣出** → 買次數有數字，交易次數 = 0（正常，留倉中）
> 2. **SELL 發生在 BUY 之前**（時間錯亂）→ FIFO 配不到，交易次數 = 0
> 3. **SELL 數量 > BUY 庫存** → 多出來的部分無從配對，不計交易次數
>
> 用這行可以直接看原始 CSV 比對：
> ```bash
> gcloud compute ssh tw-autotrader --zone=asia-east1-b \
>   --command="sudo docker exec tw_autotrader_bot python -c \"import pandas as pd; print(pd.read_csv('logs/performance.csv')[['timestamp','symbol','action','price','quantity']].to_string())\""
> ```

   🔐 **建議**：移機前先將上述檔案中真實的 Key 換成假的，移完再手動填回。

3. **資金管理**：

<a id="定期檢討排程"></a>
## 📅 定期檢討排程

自動交易不是「設好就不管」，需要定期維護。以下是建議的檢討頻率：

### 每月（20 分鐘）

- [ ] **跑「翻倍潛力股」掃瞄** — 尋找類似藥華藥（長期盤整→催化劑→翻倍）的潛力標的：
  ```bash
  cd ~/tw-autotrader
  source .venv/bin/activate
  python scripts/find_catalyst_stocks.py --top-n 20 --output-html
  ```
  打開 `img/catalyst_report_*.html`，看看有沒有新面孔。遇到不錯的潛力股，列入下次換股口袋名單（但不要急著買，等下季檢討再決定）。
- [ ] 打開 Telegram，確認上個月有正常收到交易通知
- [ ] 檢查 `performance.csv` — 至少有一兩筆交易，代表策略有在動
- [ ] 如果某策略連續一個月完全沒出手 → 放寬參數或換股票

### 每季（45 分鐘）

每季末（3/6/9/12 月最後一週）進行持股檢討與換股：

**步驟 1：跑 Grid Selector 拿推薦清單**
```bash
cd ~/tw-autotrader
source .venv/bin/activate

# 查看本季推薦持股（用 Grid Search 最佳參數）
python scripts/stock_selector_grid.py --recommend
```

**步驟 2：產出完整報告**
```bash
# 同時跑 Grid Search + 產出 HTML 報告
python scripts/stock_selector_grid.py --grid
```
打開 `img/stock_selector_grid_report.html` 查看 1728 種參數組合排名。

HTML 報告中每個參數組合的權重標示說明：

```
M:2.0  T:0.3  S:0.0  C:0.0
│     │     │     │
│     │     │     └── C = Catalyst（藥華藥模式評分）
│     │     │          越高代表越偏好「長期盤整→突破」型態
│     │     │
│     │     └── S = Stability（穩定度）
│     │           越高代表越偏好低波動股票
│     │
│     └── T = Technical（技術面）
│           越高代表越看重股價站在均線上方
│
└── M = Momentum（動能）
      越高代表越看重近季漲幅
```

例如 `M=2.0 T=0.3 S=0.0 C=0.0` 表示：動能權重 2.0 + 技術面 0.3，不看穩定度和催化劑。最佳組合通常落在純動能（M 高、其他接近 0）。

**步驟 3：對照口袋名單**
翻一下每月 catalyst 掃瞄的結果，如果有 score ≥ 60 且你研究後認同的標的，手動加入候選池。

**步驟 4：更新 .env 設定**
根據推薦清單調整 `~/tw-autotrader/.env` 中的 `PC_` 設定：
```bash
# 編輯 .env，修改對應股票的 PC_ 參數
# 例如原本持有 2330(台積電) + 2881(富邦金) + 2882(國泰金) + 6446(藥華藥)
# 如果推薦清單建議換掉其中一檔：
#   ① 刪除舊的 PC_XXX 整行
#   ② 新增新的 PC_XXX
#   ③ 調整 alloc 讓總和 = 100%
vim ~/tw-autotrader/.env
```

**步驟 5：實盤手動清倉 + 建立新部位**
被換掉的股票，在下個交易日手動賣出；新加入的股票，系統會自動在訊號觸發時進場。
